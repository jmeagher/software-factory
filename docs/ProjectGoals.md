This is a software factory that takes a project from initial idea to a fully implemented system. This will be an interactive process involving asking questions of the user requesting the software. This will include building projects from scratch, adding features to existing projects, and fixing bugs in existing projects. The implementation will be done as a set of skills, commands, and hooks that will work with both Claude Code and Cursor. The distribution will be done as plugins for those systems. See the documentation for this for [Claude Code](https://code.claude.com/docs/en/plugin-marketplaces) and for [Cursor](https://cursor.com/docs/reference/plugins). 

Key requirements that need to be covered by this:
* The implementation should be as simple as it can be to accopmlish the goals. Simplicity and clarity are more important than performance.
* Safety is important. Include guardrails to protect against harmful commands and tool invocations. These should be built-in and automatically applied to protect against losing data or breaking deployed systems.
* Validation is needed for changes. This includes making sure the code works, but also making sure it implements the features that are requsted. Red/green TDD approach should be used whenever possible to ensure behavior is correct.
* Manual validation will be needed in some cases. This should be done as soon as possible in the workflow and when other work can be done in parallel while waiting for the manual validation. This will include things like validating UIs and major changes like important API additions or changes or other changes that may impact integrations with other systems.
* This will include a base set of skills, commands, hooks, and guidance, but must also consider skills and other configuration that is available within the project being worked on.
* Clarify any ambiguities with the user before starting planning or work on code. Check any plans and output to ensure no assumptions are made that weren't in the original request or confirmed by the user.
* Break work into separate phases of work and determine ordering needs and when work can be parallelized. 
* This must be aware of context window usage as it builds. Keep the context window usage low by breaking the work into separate tasks that can be done by subagents. 
* Build a simple memory system for handling passing data between those subagents. Use exclusive locking to ensure parallel agents don't modify memory concurrently. Store the memory using the jsonl format with a single json "memory" per line in a total file. Use this for multi-agent coordination. 
* Include opentelemetry hooks for monitoring progress of the overall work. Include a main trace for the full set of work being done and extra independent traces for each significant piece of work done by the factory. These traces must include bi-directional links between them.
