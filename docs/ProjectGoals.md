This is a software factory that takes a project from initial idea to a fully implemented system. This will be an interactive process involving asking questions of the user requesting the software. This will include building projects from scratch, adding features to existing projects, and fixing bugs in existing projects. The implementation will be done as a set of skills, commands, and hooks that will work with both Claude Code and Cursor. Claude Code is the primary target; Cursor is also supported. The distribution will be done as plugins for those systems. See the documentation for this for [Claude Code](https://code.claude.com/docs/en/plugin-marketplaces) and for [Cursor](https://cursor.com/docs/reference/plugins).

Target users are experienced developers. Explanations and interactions should assume strong technical knowledge.

The scope of the factory is the requested software only. CI/CD and deployment infrastructure are assumed to already be in place unless the user explicitly includes them in scope.

Key requirements that need to be covered by this:
* The implementation should be as simple as possible to accomplish the goals. Prefer the simplest solution that satisfies the stated requirements. Do not add abstractions, helpers, or configuration for hypothetical future needs. Simplicity and clarity are more important than performance.
* Safety is important. Include guardrails to protect against harmful commands and tool invocations. These should be built-in and automatically applied:
  1. **Hooks**: Block dangerous shell commands (e.g., `rm -rf`, force-push, `git reset --hard`, destructive `DROP`/`DELETE` SQL without WHERE clauses, and similar).
  2. **Code standards**: Applied as part of the review/validation phase to catch harmful code patterns (injection vulnerabilities, credential exposure, insecure defaults, etc.).
* Validation is needed for changes. This includes making sure the code works, but also making sure it implements the features that are requested. Red/green TDD approach should be used whenever possible to ensure behavior is correct. A phase is considered complete when: (1) automated tests pass, and (2) any required manual validation has been confirmed by the user. The factory should not proceed to the next phase without both criteria being met for the current one.
* Manual validation will be needed in some cases. This should be done as soon as possible in the workflow and when other work can be done in parallel while waiting for the manual validation. This will include things like validating UIs and major changes like important API additions or changes or other changes that may impact integrations with other systems. What requires manual validation is defined by a mix of rules in this factory and rules specific to the product being worked on.
* This will include a base set of skills, commands, hooks, and guidance, but must also consider skills and other configuration that is available within the project being worked on. When project-specific skills, rules, or configuration conflict with factory defaults, project-specific configuration takes precedence. Factory defaults are the fallback when no project-specific rule exists.
* Clarify any ambiguities with the user before starting planning or work on code. Check any plans and output to ensure no assumptions are made that weren't in the original request or confirmed by the user.
* Break work into separate phases of work and determine ordering needs and when work can be parallelized. Use git commits as natural checkpoints between phases. If a phase fails, the working state can be recovered from the last committed checkpoint. The factory should commit before starting each phase when safe to do so.
* This must be aware of context window usage as it builds. Keep the context window usage low by breaking the work into separate tasks that can be done by subagents.
* Build a simple memory system for handling passing data between those subagents. Use exclusive locking to ensure parallel agents don't modify memory concurrently; file-based locking is sufficient (single-machine execution is assumed). Store the memory using the jsonl format with a single json "memory" per line in a total file. Use this for multi-agent coordination. Memory persists across sessions and is stored outside any single session context, enabling session resumption and multi-agent coordination without re-establishing context.
* Include opentelemetry hooks for monitoring progress of the overall work. Follow the [Claude Code monitoring configuration](https://code.claude.com/docs/en/monitoring-usage). The default OTLP export destination is `localhost`; any OTLP-compatible collector should work without additional configuration. Include a main trace for the full set of work being done and extra independent traces for each significant piece of work done by the factory. These traces must include bi-directional links between them.
